# Board and Column Entities and Schemas

**Type:** Task  
**Priority:** High  
**Story Points:** 3  
**Labels:** domain, entity, schema, board, column, core  
**Git Branch:** `feat/workbench-37-board-column-entities-schemas`

## Description

Define Board and Column domain entities and Zod schemas for validation. Board represents the visual configuration for ticket organization, and Column represents status columns within a board. These entities are tightly coupled and should be defined together.

## User Story

As a developer, I want Board and Column domain entities and schemas defined so that I can validate board/column data and implement board-related business logic.

## Acceptance Criteria

1. **Board Entity Definition**
   - [ ] Board type defined in `core/domain/board.schema.ts`
   - [ ] Fields: `id`, `projectId`, `createdAt`, `updatedAt`
   - [ ] All fields properly typed with TypeScript

2. **Board Zod Schema**
   - [ ] `BoardSchema` created for validating data from Supabase/external sources
   - [ ] UUID validation for `id` and `projectId`
   - [ ] Date coercion for `createdAt` and `updatedAt`

3. **Column Entity Definition**

4. **Column Zod Schema**

   - [ ] UUID validation for `id` and `boardId`

## Technical Considerations

- **Domain Layer**: Board and Column entities in `core/domain/board.schema.ts` (pure TypeScript, no external dependencies)
- **Schema Structure**: Follow existing patterns from `project.schema.ts` and `ticket.schema.ts`
- **Tight Coupling**: Board and Column in same file since they're closely related
- **UUID Validation**: Use consistent UUID validation pattern
- **Type Safety**: All types derived from Zod schemas using `z.infer<>`
- **No Business Logic**: Entity definitions only, business rules will be in separate ticket
