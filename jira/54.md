## Description

Create a reusable Input component in `presentation/components/ui/Input.tsx` with label, error state, and validation support. The component must follow project conventions and include full accessibility support with proper form labeling, error association, and ARIA attributes.

## User Story

As a developer, I want a reusable Input component with label and error state so that I can build consistent forms across the application with proper validation feedback and accessibility.

## Acceptance Criteria

1. **Component Structure**
   - [ ] Input component created in `presentation/components/ui/Input.tsx`
   - [ ] Component follows project conventions (arrow function, export default, `type` for props)
   - [ ] Component accepts props: `label`, `name`, `value`, `onChange`, `type`, `placeholder`, `error`, `helperText`, `required`, `disabled`, `ariaLabel`
   - [ ] Component uses SCSS variables from `styles/variables/*` for all styling values
   - [ ] Component SCSS file created in `src/styles/components/ui/Input.module.scss`

2. **Label Support**
   - [ ] Label element properly linked with `htmlFor` and input `id`
   - [ ] Label is visually associated with input field
   - [ ] Required indicator shown when `required` prop is true (e.g., asterisk)
   - [ ] Label text uses i18n translation keys (not hardcoded strings)

3. **Error State**
   - [ ] Error message displayed when `error` prop is provided
   - [ ] Error message uses `role="alert"` for screen readers
   - [ ] Error message is associated with input via `aria-describedby`
   - [ ] Input field has `aria-invalid="true"` when error exists
   - [ ] Error state has distinct visual styling (border color, text color)
   - [ ] Error message text uses i18n translation keys

4. **Helper Text**
   - [ ] Helper text displayed when `helperText` prop is provided (no error)
   - [ ] Helper text is associated with input via `aria-describedby`
   - [ ] Helper text uses subtle styling (different from error)
   - [ ] Helper text text uses i18n translation keys

5. **Accessibility**
   - [ ] Proper `<label>` element linked with `htmlFor` and input `id`
   - [ ] `aria-label` or `aria-labelledby` for all inputs
   - [ ] `aria-required="true"` for required fields
   - [ ] `aria-disabled="true"` for disabled inputs
   - [ ] `aria-invalid="true"` and `aria-describedby` for error states
   - [ ] Keyboard navigation: Tab key moves focus, Enter submits (if in form)
   - [ ] Focus indicators: visible focus outline
   - [ ] Accessibility IDs use centralized utilities from `shared/a11y/`

6. **Input Types**
   - [ ] Support for common input types: `text`, `email`, `password`, `number`, `tel`, `url`
   - [ ] Type-specific validation hints (optional, can be handled by parent form)
   - [ ] Type-specific keyboard on mobile devices

7. **States**
   - [ ] Default state: normal input appearance
   - [ ] Focus state: visual feedback when focused
   - [ ] Error state: distinct styling when error exists
   - [ ] Disabled state: visually distinct, non-interactive
   - [ ] Read-only state (optional): visual indication that field cannot be edited

8. **Styling**
   - [ ] All colors use SCSS variables (no hardcoded values)
   - [ ] All spacing uses SCSS variables
   - [ ] All font sizes use SCSS variables
   - [ ] Uses BEM methodology for class names (kebab-case, `__` for elements, `--` for modifiers)
   - [ ] No inline CSS
   - [ ] No `!important`
   - [ ] Consistent styling with Button component

9. **TypeScript**
   - [ ] Props typed with `type` (not `interface`)
   - [ ] Explicit return type for component
   - [ ] Proper TypeScript types for all props
   - [ ] No `any` types
   - [ ] Proper event types (`React.ChangeEvent<HTMLInputElement>`)

10. **i18n Integration**
    - [ ] All user-facing strings use `useTranslation()` hook
    - [ ] Translation keys added to `src/shared/i18n/messages/fr.json`
    - [ ] Translation keys follow naming conventions (dot notation, camelCase)
