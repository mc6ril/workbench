# Epic Entity and Schema

**Type:** Task  
**Priority:** High  
**Story Points:** 3  
**Labels:** domain, entity, schema, epic, core  
**Git Branch:** `feat/workbench-36-epic-entity-schema`

## Description

Define Epic domain entity and Zod schema for validation. Epic represents a grouping mechanism for related tickets, with progress calculation capabilities. This establishes the foundation for epic management features.

## User Story

As a developer, I want Epic domain entity and schema defined so that I can validate epic data and implement epic-related business logic.

## Acceptance Criteria

1. **Epic Entity Definition**
   - [ ] Epic type defined in `core/domain/epic.schema.ts`
   - [ ] Fields: `id`, `projectId`, `name`, `description`, `createdAt`, `updatedAt`
   - [ ] All fields properly typed with TypeScript

2. **Epic Zod Schema**
   - [ ] `EpicSchema` created for validating data from Supabase/external sources
   - [ ] UUID validation for `id` and `projectId`
   - [ ] String validation for `name` (min 1 character, non-empty)
   - [ ] Optional/nullable `description` field
   - [ ] Date coercion for `createdAt` and `updatedAt`

3. **Epic Input Schemas**
   - [ ] `CreateEpicInputSchema` for creating new epics (without id and timestamps)
   - [ ] `UpdateEpicInputSchema` for updating existing epics (all fields optional)
   - [ ] Type exports: `CreateEpicInput`, `UpdateEpicInput`

4. **Type Exports**
   - [ ] `Epic` type exported from schema file
   - [ ] All input types properly exported

## Technical Considerations

- **Domain Layer**: Epic entity in `core/domain/epic.schema.ts` (pure TypeScript, no external dependencies)
- **Schema Structure**: Follow existing patterns from `project.schema.ts` and `ticket.schema.ts`
- **UUID Validation**: Use consistent UUID validation pattern (regex or Zod's `.uuid()`)
- **Type Safety**: All types derived from Zod schemas using `z.infer<>`
- **No Business Logic**: Entity definition only, business rules will be in separate ticket
